# 天工引擎 (Tiān Gōng Engine)
## AI 代理框架工程 - Harness Engineering

---

## 🎯 命名寓意：天工开物

> **「天工」** 取自《天工开物》—— 中国古代最伟大的科技百科全书。

| 寓意 | 说明 |
|------|------|
| 🔨 **巧夺天工** | 人工的精巧胜过天然。把 AI 从「不可控的魔法」变成「可工程化的生产要素」 |
| ⚙️ **精密制造** | 不是炼丹炉式的随机产出，而是流水线式的精密制造过程 |
| 🏛️ **厚重传承** | 有文化底蕴、有历史感、可以长期使用的框架，不是一时的概念炒作 |
| 🌱 **万物生长** | 知识进化、持续迭代，越用越聪明 |

---

## 🌟 核心理念
**人类掌舵，智能体执行。**

> 构建软件仍然需要纪律，但纪律更多地体现在支撑结构上，而不是代码上。保持代码库一致性的工具、抽象和反馈回路变得越发重要。

## 🏗️ 四层架构模型

| 层级 | 解决什么问题 | 关键机制 |
|------|-------------|----------|
| **Prompt** | 单次交互质量 | 清晰表达需求，减少误解 |
| **Context** | 系统性协作 | 项目上下文、标杆代码、历史决策 |
| **Agent** | 专业化分工 | 多专家角色、互相审查、责任边界 |
| **Harness** | 完整操作系统 | Hook 拦截、验证闭环、知识进化 |

---

## 🎯 各层设计要点

### 1. Prompt Engineering
**目标**：把模糊意图转化为可执行指令
**要点**：
- 明确任务边界和输出格式
- 提供必要的约束条件
- 避免过度依赖"套路"，随模型变强边际效应递减

### 2. Context Engineering
**目标**：系统性喂给 AI 项目背景，而非单次问答
**要点**：
- **筛选**：区分信号与噪声
- **组织**：代码结构、决策记录、标杆示例的编排
- **更新**：上下文随项目演进而迭代

**实践**：
- 项目 README + 架构文档
- 标杆代码片段库
- 历史决策记录（ADR）
- 领域知识文档

### 3. Agent Engineering
**目标**：模拟专业团队的分工与制衡
**要点**：
- **专业化**：写代码的、审代码的、写文档的各司其职
- **质量控制**：输出必须经过另一角色审查
- **可追溯**：每个决策知道是哪个 agent 做的

**⚠️ 警惕陷阱**：
- ❌ 为拆而拆 → 复杂度上升，收益不增
- ❌ 通信开销 → agent 间传递信息可能丢失
- ❌ 责任分散 → 最终谁对结果负责

### 4. Harness Engineering（本层）
**目标**：设计让 AI 安全、高效、持续进化的操作系统

**核心组件**：

| 组件 | 作用 |
|------|------|
| Prompt | 单次交互质量 |
| Context | 记忆与知识传承 |
| Agent | 专业化分工 |
| Hook | 自动拦截与触发 |
| 验证闭环 | 质量门禁 |
| 知识进化 | 从结果中学习 |

**关键实践**：

#### 1️⃣ 零信任架构
- ❌ 不信任任何单一组件
- ✅ 所有输入都要验证
- ✅ 所有输出都要检查
- ✅ 所有权限都要最小化

#### 2️⃣ 自动拦截（Hook）
- ⛔ 危险操作拦截（`rm -rf`、硬编码密码等）
- 📋 合规检查
- ✅ 格式验证

#### 3️⃣ 验证闭环
- ✅ 代码必须通过测试
- ✅ 文档必须有依据
- ✅ 决策必须有记录

#### 4️⃣ 知识进化
- ✅ 验证通过的输出反哺 Context
- ✅ 失败案例沉淀为检查清单
- ✅ 持续优化 prompt 模板

## 🗺️ 核心架构原则

### 1. 代码仓库是记录系统

**❌ 反模式：一个巨大的 AGENTS.md 百科全书**
- 情境是稀缺资源，巨大的指令文件会挤掉任务、代码和相关文档
- 过多的指导反而无效：当一切都"重要"时，一切都不重要了
- 会立即腐烂：庞杂的手册会变成陈旧规则的坟场
- 难以核实：单个 blob 不适合机械检查

**✅ 正确模式：结构化 docs/ 目录 + 简短的 AGENTS.md（约 100 行）作为地图**
- 渐进式披露：智能体从小而稳定的切入点开始，按需深入
- 设计文档编目和索引，带验证状态
- 执行计划（Execution Plans）作为一等工件
- CI 自动验证知识库的新鲜度、交叉链接和结构

---

### 2. 目标是「智能体可读性」

> **从智能体的角度来看，它在运行时无法在情境中访问的任何内容都是不存在的。**

Google Docs、聊天记录、人们头脑中的知识 = 对智能体来说不存在  
代码仓库本地的、已版本化的工件 = 智能体能看到的全部

**必须放入代码仓库的内容：**
- 架构模式和设计原则
- 产品原则和工程规范
- 团队文化（包括表情符号偏好！）
- 命名约定和代码风格
- 所有业务领域知识

**技术选择策略：**
- 倾向选择「枯燥」的技术：可组合、API 稳定、训练数据里表现好
- **有时重新实现部分功能比绕过不透明的上游行为更便宜**

---

### 3. 规范架构与品味

**通过强制执行不变量，而非微观管理实施过程。**

#### 严格的分层架构示例
```
Types → Config → Repo → Service → Runtime → UI
```
- 依赖方向经过严格验证
- 仅允许有限的一组边
- 横切关注点（认证、连接器、遥测、功能标志）通过单一显式接口进入

#### 「品味不变量」示例
- ✅ 结构化日志记录模式
- ✅ 一致的命名约定（类型、函数、变量）
- ✅ 文件大小限制
- ✅ 特定平台的可靠性要求
- ✅ 在边界处解析数据形状（parse, don't validate）

> **关键洞察：**  
> 在以人为本的工作流程中，这些规则可能会让人感到迂腐或束缚。  
> 有了智能体，它们就成了 **倍增器**：一旦编码，它们就能立即应用于所有地方。

## 🚀 工作流最佳实践

### 吞吐量改变了合并的理念

| 传统工程 | Harness Engineering |
|----------|---------------------|
| 严格的合并门 | 尽量减少阻塞合并门 |
| 长 PR 生命周期 | PR 生命周期很短 |
| 偶发失败 = 阻塞 | 偶发失败 = 后续重跑解决 |
| 等待成本低，纠错成本高 | 纠错成本低，等待成本高 |

> **在智能体吞吐量远超人类注意力的系统中，纠错成本低，而等待成本高。**

### 智能体端到端自主能力（目标状态）

给定一个提示，智能体应该能够：

| 步骤 | 能力 |

## 🧹 熵与垃圾回收系统

### 问题：Codex 会复制所有模式，包括不好的模式
- 代码漂移不可避免
- 人工每周清理 "AI 残渣" = 不可扩展

### 解决方案：黄金原则 + 循环清理流程

#### 🌅 黄金原则示例
1. **更倾向使用共享实用程序包**，而不是手工编写的辅助工具，以便将不变式集中管理
2. **不使用 "YOLO 式" 探测数据** - 验证边界，或依赖类型化的 SDK，这样智能体就不会意外地基于猜测的结构进行构建
3. ...（添加你自己的原则）

#### 🔄 垃圾回收工作流
1. 定期运行后台 Codex 任务扫描偏差
2. 更新代码库质量等级
3. 发起针对性重构 Pull Request
4. 大多数可以在一分钟内完成审查并自动合并

> **技术债务就像一笔高息贷款：不断地以小额贷款的方式偿还债务，总比让债务不断累积，再痛苦地一次解决要好得多。**
>
> **人类的品味一旦被捕捉，就会持续应用于每一行代码。**

## 📋 智能体产出的完整范围

智能体应该生成 **一切**：

| 类别 | 内容 |
|------|------|
| 📦 产品 | 产品代码、单元测试、集成测试 |
| ⚙️ 基础设施 | CI 配置、发布工具、部署脚本 |
| 🔧 工具 | 内部开发者工具、脚手架、代码生成器 |
| 📚 文档 | 设计文档、API 文档、用户手册、架构决策记录 |
| 📊 评估 | 评估框架、性能基准、质量检查器 |
| 💬 代码审查 | 审阅评论模板、自动回复、反馈处理 |
| 🗂️ 仓库管理 | 管理代码仓库本身的脚本、lint 规则、格式配置 |
| 📈 可观察性 | 生产仪表板定义、告警规则、日志聚合配置 |

## 🛠️ 实现指南

### 第一步：设置知识系统

```
your-repo/
├── AGENTS.md              # 简短的入口地图 (~100 行)
├── docs/
│   ├── architecture/      # 架构文档和分层图
│   ├── principles/        # 产品和工程原则
│   ├── patterns/          # 代码模式和最佳实践
│   ├── execution-plans/   # 执行计划和进度日志
│   ├── style-guide/       # 代码风格和命名约定
│   └── taste/             # 品味不变量和 lint 规则
└── .clawhub/
    └── skills/            # 项目特定的技能
```

### 第二步：实现架构约束器

创建自定义 linter 来强制执行架构边界：
- 分层依赖检查
- 命名约定验证
- 文件大小限制
- 模式一致性检查

### 第三步：构建垃圾回收系统

- 定期扫描代码偏差
- 自动发起针对性重构
- 代码质量等级追踪
