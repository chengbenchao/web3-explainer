## 用户分享的排 Bug 方法论

**第一步：复现 — 确认"它真的是个 bug"**
- 能不能稳定复现？还是偶发？
- 复现条件是什么？（环境、数据、操作步骤）
- **很多时候"bug"其实是预期行为被误解了**，这步能筛掉一半问题

**第二步：定位 — 找到"病灶"而不是"症状"**
- 报错信息是什么？日志里有什么线索？
- 从报错往上追溯，**找到第一个出错的地方**，而不是最后一个
- 用二分法缩小范围：注释/跳过一半代码，看问题还在不在

**第三步：根因分析 — 回答"为什么会这样"**
- 这是最关键的一步。不要看到空指针就加个 null check 完事
- 问自己：**为什么会是 null？谁应该负责不为 null？**
- 常见根因类型：
  - 状态不一致（数据不同步）
  - 时序问题（竞态条件、初始化顺序）
  - 边界条件（空值、最大值、并发）
  - 依赖假设错误（API 返回格式变了、配置没生效）

**第四步：修复 + 验证**
- 修复根因，不是掩盖症状
- 写一个失败的测试来证明 bug 存在，修复后测试通过
- 考虑：这个修复会不会引入新问题？

**第五步：复盘**
- 这个 bug 能被更早发现吗？（更好的日志？更早的校验？测试覆盖？）
- 同类问题会不会在其他地方出现？

**核心原则：不要急着修，先搞清楚为什么坏。**
一个类比：医生不会看到发烧就开退烧药，他要先找到感染源。排 bug 同理
